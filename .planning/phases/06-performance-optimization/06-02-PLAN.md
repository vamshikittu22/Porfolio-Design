**Performance considerations:**
- backdrop-filter is GPU-intensive (requires compositing layer)
- Multiple overlapping glass elements can cause repaints
- Scroll-triggered animations may cause layout thrashing
- Mobile devices have less GPU power than desktop

**Gap:**
- No explicit GPU acceleration hints (transform: translateZ(0) or will-change)
- Potential over-use of backdrop-filter on static elements
- No containment properties to isolate paint/layout
- Framer Motion animations may trigger unnecessary reflows

**Requirements:**
- **PERF-01**: Glass morphism effects optimized for 60fps performance
- **PERF-04**: Chapter transitions are smooth without jank

### Task 1: Add GPU Acceleration to Glass Morphism Classes

**File:** `src/styles/glass-morphism.css`

Explicitly promote glass elements to GPU compositing layers for better performance.

**Current glass classes:**
```css
.glass-panel {
  background: rgba(var(--t-bg-el-rgb), 0.7);
  backdrop-filter: blur(12px);
  /* ... */
}

.glass-card {
  background: rgba(var(--t-bg-el-rgb), 0.5);
  backdrop-filter: blur(8px);
  /* ... */
}

.glass-overlay {
  background: rgba(var(--t-bg-el-rgb), 0.8);
  backdrop-filter: blur(16px);
  /* ... */
}
```

**Enhanced with GPU hints:**
```css
/* Glass Panel - Semi-transparent containers with moderate blur */
.glass-panel {
  background: rgba(var(--t-bg-el-rgb), 0.7);
  backdrop-filter: blur(12px);
  border: 1px solid rgba(var(--t-border-rgb), 0.3);
  box-shadow: 0 8px 32px rgba(0, 0, 0, 0.1);
  
  /* GPU acceleration */
  transform: translateZ(0);
  will-change: backdrop-filter;
  
  /* Paint containment for scroll performance */
  contain: paint;
}

/* Glass Card - Lighter background for interactive elements */
.glass-card {
  background: rgba(var(--t-bg-el-rgb), 0.5);
  backdrop-filter: blur(8px);
  border: 1px solid rgba(var(--t-border-rgb), 0.2);
  box-shadow: 0 4px 16px rgba(0, 0, 0, 0.08);
  
  /* GPU acceleration */
  transform: translateZ(0);
  will-change: transform, backdrop-filter;
  
  /* Layout containment for animation performance */
  contain: layout paint;
}

/* Glass Overlay - Stronger blur for overlays (header/footer/modals) */
.glass-overlay {
  background: rgba(var(--t-bg-el-rgb), 0.8);
  backdrop-filter: blur(16px);
  border: 1px solid rgba(var(--t-border-rgb), 0.4);
  box-shadow: 0 12px 48px rgba(0, 0, 0, 0.15);
  
  /* GPU acceleration */
  transform: translateZ(0);
  will-change: backdrop-filter;
  
  /* Strict containment for fixed positioning */
  contain: strict;
}

/* Reduced motion: disable expensive backdrop-filter */
@media (prefers-reduced-motion: reduce) {
  .glass-panel,
  .glass-card,
  .glass-overlay {
    backdrop-filter: none;
    will-change: auto;
    background: rgba(var(--t-bg-el-rgb), 0.95); /* More opaque fallback */
  }
}

/* Fallback for browsers without backdrop-filter support */
@supports not (backdrop-filter: blur(12px)) {
  .glass-panel,
  .glass-card,
  .glass-overlay {
    background: rgba(var(--t-bg-el-rgb), 0.95);
  }
}
```

**Why these properties:**
- `transform: translateZ(0)` → Forces GPU compositing layer (hardware acceleration)
- `will-change: backdrop-filter` → Hints browser to optimize for blur changes
- `contain: paint/layout/strict` → Isolates repaints to element boundaries
- Reduced motion fallback → Removes expensive blur for accessibility
- @supports fallback → Graceful degradation for older browsers

**Acceptance:**
- GPU acceleration applied to all glass classes
- Containment properties added
- Reduced motion fallback implemented
- @supports fallback verified

### Task 2: Optimize Scroll-Triggered Animations

**File:** `components/layout/ChapterHeader/ChapterHeader.tsx`

Ensure header auto-hide animation doesn't cause layout thrashing during scroll.

**Current implementation pattern:**
- Uses `useScrollDirection` hook
- Header visibility toggles based on scroll direction
- Framer Motion animates height/opacity changes

**Optimization: Use transform instead of height**

**Before (causes reflow):**
```typescript
<motion.header
  initial={{ opacity: 1, height: 'auto' }}
  animate={{ 
    opacity: isVisible ? 1 : 0,
    height: isVisible ? 'auto' : 0 
  }}
  transition={{ duration: 0.3 }}
>
```

**After (GPU-accelerated):**
```typescript
<motion.header
  initial={{ opacity: 1, y: 0 }}
  animate={{ 
    opacity: isVisible ? 1 : 0,
    y: isVisible ? 0 : -100 
  }}
  transition={{ duration: 0.3 }}
  style={{ transform: 'translateZ(0)' }}
>
```

**Check files:**
- `components/layout/ChapterHeader/ChapterHeader.tsx`
- `components/layout/ChapterFooter/ChapterFooter.tsx`
- `components/layout/ChapterSidebar/ChapterSidebar.tsx`

**If current implementation uses height/width animations, replace with transform (translateX/translateY).**

**Acceptance:**
- Header/footer animations use transform instead of height
- No layout shifts during scroll
- Animations remain smooth at 60fps

### Task 3: Debounce Scroll Events in useScrollDirection Hook

**File:** `src/hooks/useScrollDirection.ts`

Add debouncing to scroll event listener to reduce CPU overhead.

**Current implementation (if exists):**
```typescript
useEffect(() => {
  const handleScroll = () => {
    const currentScrollY = window.scrollY;
    
    if (currentScrollY > lastScrollY.current && currentScrollY > threshold) {
      setScrollDirection('down');
    } else if (currentScrollY < lastScrollY.current) {
      setScrollDirection('up');
    }
    
    lastScrollY.current = currentScrollY;
  };

  window.addEventListener('scroll', handleScroll);
  return () => window.removeEventListener('scroll', handleScroll);
}, [threshold]);
```

**Optimized with requestAnimationFrame:**
```typescript
useEffect(() => {
  let ticking = false;

  const handleScroll = () => {
    if (!ticking) {
      window.requestAnimationFrame(() => {
        const currentScrollY = window.scrollY;
        
        if (currentScrollY > lastScrollY.current && currentScrollY > threshold) {
          setScrollDirection('down');
        } else if (currentScrollY < lastScrollY.current) {
          setScrollDirection('up');
        }
        
        lastScrollY.current = currentScrollY;
        ticking = false;
      });
      
      ticking = true;
    }
  };

  window.addEventListener('scroll', handleScroll, { passive: true });
  return () => window.removeEventListener('scroll', handleScroll);
}, [threshold]);
```

**Why this helps:**
- `requestAnimationFrame` syncs scroll handler with browser repaint cycle
- Prevents scroll handler from running multiple times per frame
- `{ passive: true }` hint improves scroll performance
- Reduces CPU usage during rapid scrolling

**Acceptance:**
- useScrollDirection hook uses requestAnimationFrame
- Scroll listener marked as passive
- No performance regression in scroll responsiveness

### Task 4: Lazy Load Heavy Framer Motion Animations

**File:** `components/landing/LandingPage.tsx`

Defer Framer Motion animations on landing page until after initial render.

**Current pattern (immediate animation):**
```typescript
<motion.div
  initial={{ opacity: 0, y: 20 }}
  animate={{ opacity: 1, y: 0 }}
  transition={{ delay: index * 0.1 }}
>
  {/* Chapter card */}
</motion.div>
```

**Optimized (conditional animation):**
```typescript
const [shouldAnimate, setShouldAnimate] = useState(false);

useEffect(() => {
  // Defer animations until after initial paint
  const timer = setTimeout(() => setShouldAnimate(true), 100);
  return () => clearTimeout(timer);
}, []);

return (
  <motion.div
    initial={shouldAnimate ? { opacity: 0, y: 20 } : false}
    animate={shouldAnimate ? { opacity: 1, y: 0 } : {}}
    transition={{ delay: index * 0.1 }}
  >
    {/* Chapter card */}
  </motion.div>
);
```

**Alternative: Disable animations on slow devices**
```typescript
const prefersReducedMotion = window.matchMedia('(prefers-reduced-motion: reduce)').matches;
const isSlowDevice = navigator.hardwareConcurrency <= 2; // Low CPU cores

const shouldAnimate = !prefersReducedMotion && !isSlowDevice;
```

**Acceptance:**
- Landing page animations deferred by 100ms
- Animations respect reduced motion preference
- Optional: Detect slow devices and disable stagger animations
- No visual regression for users with modern devices

## Notes

- GPU acceleration via `transform: translateZ(0)` is a well-established pattern
- `will-change` should be used sparingly (only on animating elements) to avoid memory overhead
- `contain` property is powerful but less supported (Safari 15.4+, Chrome 52+)
- requestAnimationFrame is critical for scroll-triggered animations
- Reduced motion support improves accessibility AND performance
- Performance budgets prevent regression in future development
- Mobile devices struggle more with backdrop-filter than desktop
- HTTP/2 parallel chunk loading (from 06-01) significantly helps perceived performance

**Key insight:**
> Glass morphism (backdrop-filter) is expensive, but unavoidable for the design. Strategic GPU acceleration + containment + reduced motion fallbacks make it performant enough for 60fps on modern devices.
